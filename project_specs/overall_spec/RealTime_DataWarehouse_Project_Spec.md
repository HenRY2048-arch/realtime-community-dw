          内容社区实时数仓项目 (Real-Time Data Warehouse) 核心规约
==========================================================================

一、 项目定义与目标
--------------------------------------------------------------------------
本项目是一个针对"内容社区"场景构建的实时数据处理系统。它通过模拟海量用户的交互行为，
利用 Spark Structured Streaming 框架实现数据的实时采集、多维关联、质量检测与聚合分析。

核心目标：
1. 业务逻辑验证：实现高度仿真的社区行为流（含推荐逻辑、会话管理）。
2. 计算链路开发：构建基于 Event Time 的窗口聚合体系。
3. 架构模式探索：采用"伪 Lambda"架构，实时层负责计算与展示，离线层（Delta Lake）
   负责归档与口径统一。

二、 核心业务逻辑架构
--------------------------------------------------------------------------
1. 分发算法 (70/30 策略)：
   - 兴趣匹配 (70%)：实时行为优先与用户维表中的 interest_tags 关联。
   - 随机探索 (30%)：系统随机从全量内容库抽样。
   - 详见：RealTime_Mock_Generator_Design_Doc.md -> "3. 核心算法"

2. 事件语义与生命周期 (13 种事件深度解析)：
   - 【基础触发】app_open: 会话起点，强制刷新 session_id。
   - 【曝光路径】enter_item: 用户点击笔记详情页，记录开始时间（权重 40%）。
   - 【消费反馈】leave_item: 用户退出详情页，需计算停留时长 duration_sec（权重 35%）。
   - 【互动反馈】like, fav, share, dislike: 对笔记内容的显性态度表达（总权重 17%）。
   - 【社交意向】dm (私信), follow, unfollow: 用户间的社交影响力转化信号。
   - 【行为辅助】click_tag, enter_profile, comment: 深度探索行为。
   - 详见：RealTime_Mock_Generator_Design_Doc.md -> "五、 核心业务逻辑与数据字典"

3. 会话管理 (Session Tracking)：
   - 格式：sess_{timestamp}_{user_id}
   - 逻辑：支持 app_open 触发及 30 分钟无活动自动超时。
   - 详见：RealTime_Mock_Generator_Implementation_Doc.md -> "第三阶段：会话管理"

三、 混沌工程与数据挑战
--------------------------------------------------------------------------
数仓逻辑具备以下人为注入的挑战处理能力，以验证系统的健壮性：

1. 时间语义异常：
   - B1 乱序数据：10% 事件带有 30-90s 延迟，要求设置合理的 Watermark。

2. 业务行为异常：
   - A1 刷屏评论：同一用户对同一笔记在60s滑动窗口内连续发送 ≥5 条评论，且满足时间紧邻（间隔 ≤10s）与（窗口高频 ≥5条 或 重复内容 ≥4条相同）的双重判定。
   - A2 疯狂点赞：单用户 5s 内对 15 篇不同笔记疯狂点赞。
   - 处理：必须打上 "is_cheat" 标签并保留痕迹。

3. 质量挑战异常：
   - D1 秒退检测：停留时间极短（0.2s - 1.2s）的无效曝光。
   - E1 超速滑动：停留时间短但内容极长（content_len > 2000）的滑动行为。
   - C 异常挂机：停留时间超长（1800s - 3600s）的非正常阅读。

4. 流量整形挑战：
   - 洪峰流量：每 10 分钟突增至 150-200 EPS，验证背压处理。

   - 详见：RealTime_Mock_Generator_Design_Doc.md -> "5. 异常注入"
   - 详见：RealTime_Mock_Generator_Implementation_Doc.md -> "第四阶段：异常注入引擎"

四、 技术架构与存储规约
--------------------------------------------------------------------------
1. 拓扑结构：
   Mock Generator (Java 8) -> Kafka (Topic: content_events) -> Spark (Scala 2.12)

2. 存储介质 (Sink)：
   - 实时层：Redis (存储高频更新的 DWS 指标)。
   - 归档层：Delta Lake (作为数据湖进行结构化归档，确保数据一致性与回溯能力)。

3. 维表管理 (Join Strategy)：
   - 模式：Stream-Static Left Join。
   - 容错：关联失败显示为 "Unknown"，严禁丢失数据。
   - 特性：必须支持维表热更新（无需重启 Spark 即可感知 CSV 变更）。

4. 运行规格：
   - 本地开发并行度限制：Local[2]。
   - 优先级：优先保证低延迟，宁愿增加计算资源。

五、 FAQ
--------------------------------------------------------------------------
Q: 发现疑似作弊用户后直接丢弃吗？
A: 不。必须保留作弊痕迹，在流中增加标记字段，后续在业务侧进行过滤。

Q: 维表关联不到 ID 怎么办？
A: 标记为 Unknown。数据完整性高于一切，任何情况下的 Drop Data 都是严重事故。

Q: 为什么离线层只负责归档？
A: 为了保证数据口径的绝对统一，通过 Delta Lake 进行归档，牺牲部分实时性以换取高一致性。
